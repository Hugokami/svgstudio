## 2024-05-24 - Unthrottled DOM reads during drag operations
**Learning:** `svgviewer.html` is a large single-file application where layout thrashing is a primary performance bottleneck. Several drag operations (marquee selection, panning, scrubbing) were recalculating layout (e.g., `getBoundingClientRect`) or modifying DOM state inline during `mousemove` events without `requestAnimationFrame` throttling, causing unnecessary layout calculation on every frame.
**Action:** When working with custom drag/drop or mousemove-driven interactions in this specific file, always implement `requestAnimationFrame` and ensure bounding rects are cached on `mousedown` rather than calculated on `mousemove`.

## 2024-05-25 - Eliminating UI Input Lag with Caching and rAF Throttling
**Learning:** Heavy layout-recalculating mouse and pointer interactions (like window resizing, timeline scrubbing, and marquee selection) in `js/app.js` and `svgviewer.html` were not utilizing `requestAnimationFrame` (rAF). They were blocking the main thread during high-frequency move events and forcing unnecessary synchronous DOM reads (e.g., `getBoundingClientRect()`, `clientWidth`), which can result in significant layout thrashing on heavy SVG pages.
**Action:** When implementing or modifying custom drag and drag-like selection handlers, cache any DOM position/dimension reads directly in the `mousedown` / `pointerdown` phase. Perform updates to properties and DOM styling exclusively within a `requestAnimationFrame` wrapper during `mousemove`, and explicitly ensure proper cleanup by cancelling pending frames during `mouseup`.
