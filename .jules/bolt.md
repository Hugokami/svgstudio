## 2024-05-24 - Unthrottled DOM reads during drag operations
**Learning:** `svgviewer.html` is a large single-file application where layout thrashing is a primary performance bottleneck. Several drag operations (marquee selection, panning, scrubbing) were recalculating layout (e.g., `getBoundingClientRect`) or modifying DOM state inline during `mousemove` events without `requestAnimationFrame` throttling, causing unnecessary layout calculation on every frame.
**Action:** When working with custom drag/drop or mousemove-driven interactions in this specific file, always implement `requestAnimationFrame` and ensure bounding rects are cached on `mousedown` rather than calculated on `mousemove`.

## 2025-02-14 - JIT Compilation Overhead in High-Frequency Animations
**Learning:** In the expressions evaluation block for SVG animations (`js/app.js` and `svgviewer.html`), dynamic math expressions like `sin(t)` are evaluated using `new Function(...)`. Previously, this happened dynamically on every frame during the animation loop via GSAP tickers. Creating a `new Function` on every frame destroys JavaScript engine JIT (Just-In-Time) compilation and severely harms runtime performance, causing frame drops and increased CPU usage.
**Action:** When dynamically interpreting user-defined expressions inside high-frequency event loops like GSAP ticker `onUpdate` or `requestAnimationFrame`, always construct the function strings, compile them using `new Function(...)` once, and cache them in a `Map` keyed by the expression string. Then, call the cached function in the loop.
