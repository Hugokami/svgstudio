## 2024-05-24 - Unthrottled DOM reads during drag operations
**Learning:** `svgviewer.html` is a large single-file application where layout thrashing is a primary performance bottleneck. Several drag operations (marquee selection, panning, scrubbing) were recalculating layout (e.g., `getBoundingClientRect`) or modifying DOM state inline during `mousemove` events without `requestAnimationFrame` throttling, causing unnecessary layout calculation on every frame.
**Action:** When working with custom drag/drop or mousemove-driven interactions in this specific file, always implement `requestAnimationFrame` and ensure bounding rects are cached on `mousedown` rather than calculated on `mousemove`.
## 2024-05-24 - Dynamic expression evaluation inside GSAP ticker loops
**Learning:** `js/app.js` and `svgviewer.html` make heavy use of dynamic math expressions embedded as custom attributes (`data-expr-*`). These were being lazily evaluated inside GSAP's high-frequency tick handlers (`updateTimelineUI` at 60fps) using `new Function()`. V8 and other JS engines cannot optimize these dynamic calls and spend significant time JIT-compiling the exact same string every frame, causing a severe frame-rate drop (e.g. 100ms per 100k calls vs 3ms).
**Action:** Always cache the compiled result of `new Function()` inside a `Map` keyed by the expression string (`window.expressionCache = window.expressionCache || new Map()`) to skip recompilation for subsequent animation ticks. Additionally, iterating over `Element.attributes` using `Array.from()` allocates a new array each frame; use a standard `for` loop with `.length` and `.item(i)` to iterate DOM attributes in hot paths without triggering garbage collection overhead.
