## 2024-05-24 - Unthrottled DOM reads during drag operations
**Learning:** `svgviewer.html` is a large single-file application where layout thrashing is a primary performance bottleneck. Several drag operations (marquee selection, panning, scrubbing) were recalculating layout (e.g., `getBoundingClientRect`) or modifying DOM state inline during `mousemove` events without `requestAnimationFrame` throttling, causing unnecessary layout calculation on every frame.
**Action:** When working with custom drag/drop or mousemove-driven interactions in this specific file, always implement `requestAnimationFrame` and ensure bounding rects are cached on `mousedown` rather than calculated on `mousemove`.
## 2025-02-24 - Throttling rapid UI interactions using requestAnimationFrame correctly
**Learning:** Naively using `if (myRAF) return; myRAF = requestAnimationFrame(...)` to throttle `mousemove` drops pointer coordinates and leads to UI rendering exactly one frame behind the actual pointer. Using `cancelAnimationFrame` inside `mouseup` compounds the problem by deleting the final frame entirely, causing UI states to permanently desync from the drop zone.
**Action:** When implementing RAF throttles for `mousemove` and `pointermove` operations (like drags or resizing), ALWAYS capture the latest `e.clientX / e.clientY` values into external variables outside the RAF callback. Have the RAF consume these external variables, and strictly force a synchronous final execution block on `pointerup` before dropping the RAF reference.
