## 2024-05-24 - Unthrottled DOM reads during drag operations
**Learning:** `svgviewer.html` is a large single-file application where layout thrashing is a primary performance bottleneck. Several drag operations (marquee selection, panning, scrubbing) were recalculating layout (e.g., `getBoundingClientRect`) or modifying DOM state inline during `mousemove` events without `requestAnimationFrame` throttling, causing unnecessary layout calculation on every frame.
**Action:** When working with custom drag/drop or mousemove-driven interactions in this specific file, always implement `requestAnimationFrame` and ensure bounding rects are cached on `mousedown` rather than calculated on `mousemove`.
## 2024-05-24 - DOM Selector Optimization for SVG class discovery
**Learning:** When using `jsdom` to benchmark DOM selection performance, the results can be misleading. `jsdom`'s JS-based implementation can be significantly slower than a browser's native C++ engine. However, in an actual browser environment, querying `querySelectorAll('[class*="anim-"]')` pushes the initial filtering to the heavily-optimized C++ layer, dramatically reducing overhead compared to `querySelectorAll('*')` and subsequent JavaScript array mapping/filtering for large SVG documents.
**Action:** When working with large DOM structures (like complex SVGs), always prefer specific attribute selectors over the universal `*` selector to delegate work to the native rendering engine, even if JS benchmarks using `jsdom` suggest otherwise.
